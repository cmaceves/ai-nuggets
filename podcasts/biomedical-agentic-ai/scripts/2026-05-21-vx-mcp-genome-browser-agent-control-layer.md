## Script

Today's nugget is a fresh bioRxiv preprint from the Shirokikh group at the University of Western Australia describing VX, a desktop genome and transcriptome browser whose first-class users are LLM agents, not humans. The framing is what makes it interesting. Today's genome browsers — IGV, JBrowse, UCSC — were all designed around a human at a graphical interface, with at most a thin scripting hook bolted on the side. As language model assistants become routine in bioinformatics, that design creates a hard bottleneck. An agent can't see what the browser is showing, can't drive it through the same operations the human uses, and ends up reasoning about a completely separate ecosystem of command-line tools that don't talk to the viewer at all. The agent and the human are in different worlds.

VX flips that. It exposes its full functionality through two endpoints. There's an embedded HTTP API on the loopback interface, and there's a Model Context Protocol server with thirty-nine tools right now. Both serve exactly the same contract that drives the GUI. Loading data, navigating to a locus, managing tracks, running analyses, capturing figures — all of it. The same operation a user clicks through is the operation an agent invokes through MCP. There is no agent-specific subset or limited scripting dialect. The full tool surface is the agent surface. The authors call this a native agent-control layer, and that phrase is doing a lot of work — it's saying the integration is structural, not a wrapper.

Underneath the agent layer, VX bundles an integrated analysis framework with more than fifty analyses spanning signal processing and peak calling, quantification, variant analysis, alignment statistics, and cross-track comparisons. Results are written to disk and, where appropriate, automatically loaded back as new tracks. There is a four-level scope hierarchy running from the current viewport up to the whole dataset, so an agent can ask for the same analysis at the resolution that matches what it's reasoning about. That tight loop — analyze, materialize results as a track, look at them, analyze again — is the loop a human power user already does between a browser and a command-line tool, except now it happens entirely inside one program and an agent can drive it.

There's a second contribution worth pulling out, which is first-class transcript-coordinate support. Ribosome profiling and direct RNA sequencing both produce data that is naturally indexed in transcript space, not chromosome space, and existing chromosome-oriented viewers handle this awkwardly. VX treats transcript-scale and genome-scale data in a unified interface, which makes it one of the better matches I've seen for translation-level and isoform-level work specifically. That's an audience that has been historically underserved by mainstream browsers.

The engineering choices are also worth noting because they shape what's possible. VX is written in D using GTK 3 and OpenGL — unusual for bioinformatics but consistent with the goal of a fast native desktop tool that can stream and render large datasets without web-frontend overhead. The HTTP API and the MCP protocol are both fully specified in the paper itself, which means third-party clients — including specialized agents — can be built against the spec without depending on the core implementation. That openness matters: it's the difference between an agent integration that one team owns and an agent integration that anyone can extend.

Two things to like about this from an agentic-AI perspective. First, the architectural commitment. Most agent-friendly scientific tools today expose a narrow JSON API that covers maybe ten percent of what the GUI can do, which means agents end up systematically less capable than human users. VX inverts that — the GUI is one client of the same protocol the agent uses, not a privileged interface with hidden capabilities. Second, the choice of MCP as the agent-facing contract is a bet that the protocol is going to be a durable standard rather than a transient one, and that bet looks better every month as more tools converge on it. A genome browser that speaks MCP is composable with any agent runtime that speaks MCP, with no glue code.

One caveat. The paper specifies the protocol but doesn't include a full agentic case study showing what end-to-end discovery looks like with an LLM driving VX over MCP — that's left as future work. So the architectural argument is fully made, but the empirical proof that agents using VX do better, faster, or more reliable analysis than the same agents with shell access alone is not in this preprint. That study is what would turn this from a well-designed substrate into a demonstrated workflow.

Read it for the design philosophy. The takeaway worth keeping is the principle: when you're building scientific software in 2026, the agent-facing interface shouldn't be a wrapper around the GUI — they should be the same interface, with the GUI as one of several clients.
